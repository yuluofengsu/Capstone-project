## 2. 10分钟详细陈述稿（可直接读）

各位老师好，汇报题目是《社区化图书共享与推荐平台的设计与实现》。

本课题的研究背景是：传统图书管理系统通常更关注“书籍管理”和“借还登记”，而在用户体验层面仍存在不足，例如找书效率不高、个性化推荐能力弱、读者之间缺少交流场景。针对这些问题，本项目构建了一个集图书借阅、个性化推荐、社区互动于一体的平台，目标是提升读者获取图书的效率与阅读参与度。

系统在架构上采用前后端分离。前端基于 Vue2 和 Ant Design Vue 实现界面与交互逻辑，通过 Vuex 管理全局状态；后端基于 Django4.1.4 与 Django REST Framework 对外提供 RESTful 接口；数据库默认使用 SQLite，兼容 MySQL。请求通信通过 Axios 统一封装，前端在请求拦截器中自动注入用户 token 与管理员 token，并在响应阶段处理 403 鉴权失败，分别触发前台或后台退出逻辑。

系统功能主要分为五个模块。

第一，用户管理模块。该模块支持注册登录、个人资料维护、密码修改，以及管理员侧的用户查询、冻结与删除等操作，满足平台基础身份管理需求。

第二，图书管理模块。该模块包含图书信息维护、分类与标签管理、图书详情展示、评论、收藏与心愿单等功能，形成“管理端维护 + 用户端消费”的完整闭环。

第三，借阅管理模块。该模块覆盖借书、还书、延期、借阅记录查询等核心流程。系统在后端层面对库存、借阅状态和时间字段进行约束，避免非法借阅和状态不一致问题。

第四，社区互动模块。该模块支持帖子发布、点赞、评论以及阅读活动，增强读者之间的交流，提升平台活跃度与粘性。

第五，推荐模块。推荐模块基于用户行为构建用户-图书矩阵，并通过协同过滤思想计算相似用户，生成个性化图书推荐。当前行为特征主要来源于收藏与心愿数据，能够在一定程度上改善“找不到感兴趣图书”的问题。

从数据结构看，系统围绕 b_user、b_book、b_borrow、b_comment、b_classification、b_community_post 等核心表组织业务关系。日志层面包含 b_login_log、b_op_log、b_error_log，用于行为审计、操作追踪与异常定位，增强了系统可维护性。

在实现层面，项目有三点较关键。第一，请求链路统一。前端 Axios 封装减少了重复代码，保证了 token 逻辑一致性。第二，路由分层清晰。后端以 /admin 和 /index 分离管理端与用户端接口，职责边界明确。第三，业务规则后置。关键校验放在后端执行，避免仅依赖前端造成的绕过风险。

系统完成后进行了功能测试，覆盖用户管理、图书管理、分类管理、借阅管理和社区互动等核心模块。测试结果表明，主要业务流程均可稳定执行，满足预期目标。

本项目的工作结果可以概括为三点：
1) 完成了社区化图书共享与推荐平台的端到端实现；
2) 构建了可扩展的前后端分离工程结构；
3) 在推荐与社区融合方向完成了可运行验证，为后续优化提供了基础。

以上为汇报内容，恳请各位老师批评指正。

## 4. 高频问题与答案（扩展版，20题）

## 4.1 选题与意义

**Q1：为什么做这个题目？**

A：图书借阅场景中“找书难、推荐弱、互动弱”是常见问题。本项目把借阅业务、推荐能力和社区功能结合，提升了系统实用性。

**Q2：项目与普通图书管理系统的区别是什么？**

A：区别在于不只做后台管理，还面向读者提供推荐与社区互动，形成“管理+服务”双端能力。

## 4.2 技术选型

**Q3：为什么不用Spring Boot？**

A：本科项目优先考虑开发效率与可控性。Django自带ORM与后台能力，配合DRF能快速构建接口，适合毕设周期。

**Q4：为什么前端选Vue2而非Vue3？**

A：项目起步时Vue2生态更稳定，团队熟悉度更高，能保证交付质量。结构上已模块化，后续可迁移。

## 4.3 系统设计

**Q5：前后端分离的价值是什么？**

A：解耦开发、接口可复用、后续多端接入更方便，维护成本更低。

**Q6：为什么要区分admin和index路由？**

A：便于权限隔离与职责拆分，降低误调用风险，也方便后续扩展和审计。

## 4.4 数据库与业务

**Q7：Borrow表核心字段作用？**

A：`status`标识借阅状态，`borrow_time`记录借出时间，`expect_time`记录应还时间，`return_time`记录归还时间，`delayed`标记是否延期。

**Q8：如何避免非法借阅？**

A：后端统一校验库存、状态与用户身份，前端只做交互提示，关键规则不依赖前端。

**Q9：为什么需要日志三件套（登录/操作/错误）？**

A：登录日志用于安全审计，操作日志用于追踪变更，错误日志用于快速定位问题。

## 4.5 推荐算法

**Q10：推荐依据是什么行为？**

A：主要依据收藏和心愿单行为构建用户偏好，后续可加入借阅时长、评分、评论情感等特征。

**Q11：当前推荐算法局限在哪里？**

A：对新用户有冷启动问题，对行为稀疏场景效果受限，后续可结合热门推荐与内容推荐。

**Q12：推荐结果如何评估？**

A：可从点击率、借阅转化率、用户停留时长等指标评估，当前版本完成了功能可用性验证。

## 4.6 安全与性能

**Q13：token失效如何处理？**

A：前端拦截 403，根据 `AUTH_FAIL_FRONT` 或 `AUTH_FAIL_END` 分别执行用户/管理员退出并刷新。

**Q14：系统并发上来后怎么做？**

A：优先引入Redis缓存热点数据，优化SQL与索引，必要时做读写分离与异步任务处理。

## 4.7 测试与工程

**Q15：主要测试了哪些内容？**

A：用户、图书、分类、借阅、社区模块功能流程；重点测试鉴权、借还状态和互动功能。

**Q16：有没有做异常测试？**

A：做了典型异常分支验证，如登录失败、权限不足、库存不足、无效参数等场景。

**Q17：代码可维护性如何保证？**

A：前后端按模块分层，接口按admin/index域拆分，通用请求逻辑集中在request封装中。

## 4.8 论文与工作量

**Q18：你的主要工作是什么？**

A：需求分析、系统设计、后端接口开发、前端页面联调、测试文档整理与答辩材料准备。

**Q19：项目难点是什么？**

A：鉴权链路统一、借阅状态一致性、推荐算法可用性验证。

**Q20：如果再做两周会改什么？**

A：优先做推荐评估指标闭环、缓存优化、接口文档标准化。

## 6. 易被追问的“敏感点”与应对

1. **为什么密码字段看起来是明文？**
   - 回答：生产环境应采用哈希存储与加盐策略，当前版本在教学场景完成流程验证，后续会强化安全方案。

2. **推荐算法是否上线到主流程？**
   - 回答：当前完成算法模块实现与可运行验证，后续将进一步打通业务页面并完善评估指标。

3. **为什么有SQLite和MySQL两种说法？**
   - 回答：开发默认SQLite便于快速部署，生产建议切换MySQL提升数据管理与扩展能力。

4. **如何证明不是“只做了页面”？**
   - 回答：可展示后端路由、数据模型、日志表设计和关键接口响应。

## 7. 答辩礼仪与表达技巧（本科实用）

- 语速控制：每分钟约220~260字。
- 视线：讲重点时看老师，不要一直看屏幕。
- 回答结构：**先结论，再理由，最后补充**。
- 不会时：
  - “该点当前版本未完整覆盖，已完成的部分是……，后续计划是……”。
- 禁忌：
  - 不与老师争辩；
  - 不说“这个不重要”；
  - 不把问题推给“队友/别人”。
