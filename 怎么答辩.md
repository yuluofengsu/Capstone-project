# 社区化图书共享与推荐平台答辩指南（本科版）

## 0. 使用说明
- 目标：用于中国本科毕业答辩，包含**答辩介绍稿 + 常见问题与回答 + PPT设计 + 现场流程**。
- 适配项目：本仓库代码（Vue2 + Ant Design Vue + Django4.1.4 + DRF + SQLite/MySQL，含社区与推荐模块）。
- 建议时长：**陈述8分钟 + 问答7分钟**（可按学院要求调整）。

## 1. 答辩当天标准流程（中国本科常见）

1. **入场与设备检查（提前20分钟）**
   - 打开PPT、关闭消息弹窗、确认字体不乱。
   - 准备“演示兜底方案”：截图版流程（避免现场网络/服务异常）。

2. **自我介绍（15~20秒）**
   - 例：各位老师好，学生XXX，论文题目为《社区化图书共享与推荐平台的设计与实现》，下面开始汇报。

   - 一般3~6个问题，优先回答“做了什么、为什么这么做、效果如何”。

5. **结束致谢（10秒）**
   - 例：汇报完毕，恳请各位老师批评指正。

## 2. 可直接照读的“8分钟答辩介绍稿”

各位老师好，学生XXX，汇报题目是《社区化图书共享与推荐平台的设计与实现》。

本课题的出发点是：传统图书借阅系统通常只解决“借和还”，对读者个性化推荐、读者互动交流支持不足。为此，本项目实现了一个“借阅+推荐+社区”一体化平台，提升读者找书效率和阅读参与度。

系统采用前后端分离架构。前端使用 Vue2、Ant Design Vue、Vuex、Axios；后端使用 Django4.1.4 和 Django REST Framework；数据库开发环境使用 SQLite，也支持切换到 MySQL。系统通过 RESTful API 通信，前端在请求拦截器中自动携带用户 token 与管理员 token，实现前后台鉴权分离。

本系统主要实现了五类核心能力。第一是用户与权限能力，包括用户注册登录、个人信息维护、后台用户管理。第二是图书业务能力，包括图书、分类、标签、评论、收藏与心愿单管理。第三是借阅能力，包括借书、还书、延期、借阅记录查询以及后台借阅管理。第四是社区能力，包括帖子发布、点赞、评论和阅读活动。第五是推荐能力，基于用户行为构建用户-图书矩阵，利用协同过滤思想计算用户相似度，为用户推荐其未接触但潜在感兴趣的图书。

在数据设计方面，系统围绕 b_user、b_book、b_borrow、b_comment、b_classification、b_community_post 等核心表构建业务闭环。日志层面包含登录日志、操作日志、错误日志，便于审计和排障。

在关键实现方面，前端通过统一请求封装处理 token 注入和 403 失效重登；后端按 admin 与 index 两套路由分离管理接口与用户接口。借阅业务中，围绕库存、借阅状态、应还时间与延期标记进行约束，确保流程完整。社区业务支持帖子与评论双层互动，提升读者活跃度。

系统完成后进行了功能测试，覆盖用户管理、图书管理、分类管理、借阅管理、社区互动等主要模块，验证结果与预期一致，系统能够稳定完成核心流程。

本项目的主要价值在于：第一，实现了图书借阅与推荐、社区互动的融合；第二，形成了可扩展的前后端分离工程结构；第三，具备继续优化的空间，例如推荐算法精度提升、缓存加速与高并发能力增强。

以上是本次汇报内容，汇报完毕，恳请各位老师批评指正。

## 3. PPT设计（可直接制作）

- 建议 **12页**，总时长8分钟。
- 每页约35~45秒。

## 3.2 每页标题与内容

### 第1页：封面（15秒）
- 题目：社区化图书共享与推荐平台的设计与实现
- 学生姓名、学号、专业、指导教师、日期

### 第2页：研究背景与意义（40秒）
- 传统借阅系统痛点：仅借还、推荐弱、互动弱
- 本课题目标：借阅效率 + 个性化推荐 + 社区交流

### 第3页：需求分析（45秒）
- 用户端：注册登录、图书检索、借还延期、评论、发帖
- 管理端：图书/分类/用户/借阅/日志管理

### 第4页：总体架构（45秒）
- 前端：Vue2 + Ant Design Vue + Vuex + Axios
- 后端：Django + DRF
- 数据库：SQLite/MySQL
- 通信方式：RESTful API

### 第5页：数据库与核心实体（45秒）
- 重点表：b_user、b_book、b_borrow、b_comment、b_classification、b_community_post
- 关系说明：用户-图书（收藏/心愿/借阅）、帖子-评论

### 第6页：核心功能一：图书与借阅（45秒）
- 图书列表、详情、分类标签、收藏/心愿
- 借书、还书、延期、借阅记录
- 关键约束：库存、状态、应还时间

### 第7页：核心功能二：社区互动（35秒）
- 发帖、点赞、评论、活动

### 第8页：推荐算法实现（50秒）
- 思路：协同过滤（基于用户行为）
- 行为建模：收藏、心愿等行为转为评分
- 相似度计算与推荐输出流程
- 输出：未交互图书的推荐列表

### 第9页：关键工程实现（40秒）
- 请求拦截器统一注入 token
- 403统一处理与前后台分别退出
- 后端 admin/index 双路由分层
- 日志机制：登录、操作、错误

### 第10页：系统测试与结果（45秒）
- 覆盖模块：用户、图书、分类、借阅、社区
- 结果：核心流程通过，满足预期功能
- 可补1张表格截图（来自系统测试文档）

### 第11页：创新点与不足（40秒）

## 4. 答辩高频问题与标准回答（可直接背）

## 4.1 技术选型类

**Q1：为什么选 Vue2 + Django，而不是 Spring Boot 或 Vue3？**

A：本项目优先考虑开发周期、学习成本和稳定性。Vue2与Django生态成熟，组件与文档完善，能够在本科毕设周期内稳定落地。前后端分离也保证了后续可迭代性，后期可平滑升级到Vue3或替换后端框架。

**Q2：为什么使用前后端分离？**

A：前后端分离可以让界面开发和业务接口解耦，提升开发效率。接口复用性更好，后期接入小程序或移动端成本更低。

## 4.2 数据与业务类

**Q3：借阅流程如何保证数据一致性？**

A：借阅核心受库存和状态约束。借书时校验库存并写入借阅记录，还书时更新还书时间与状态。通过后端统一校验避免前端绕过规则。

**Q4：为什么要设计日志表？**

A：登录日志、操作日志、错误日志分别用于安全审计、行为追踪和异常排查。答辩中可以证明系统不仅“能用”，还“可维护”。

## 4.3 推荐算法类

**Q5：推荐算法的核心思路是什么？**

A：将用户收藏、心愿等行为映射为用户-图书矩阵，计算用户间相似度，找到与目标用户相近的用户，再推荐其未接触图书。属于协同过滤思想。

**Q6：冷启动如何处理？**

A：新用户缺少行为数据时，可先推荐热门图书、最新上架图书、分类热门图书，待行为积累后再切换到个性化推荐。

## 4.4 测试与可扩展类

**Q7：系统做了哪些测试？**

A：围绕用户、图书、分类、借阅、社区等模块设计了功能用例，验证核心流程与预期一致。重点测试了登录鉴权、借还流程、帖子互动等高频路径。

**Q8：系统还有哪些不足？**

A：推荐精度还有提升空间；高并发场景下可通过Redis缓存、读写分离继续优化；可增加更细粒度权限控制。

## 5. 现场演示脚本（2~3分钟，可选）

1. 登录用户端（展示图书首页）
2. 搜索一本书并进入详情
3. 执行借阅或加入心愿单/收藏
4. 打开“我的借阅”展示状态变化
5. 进入社区发布或查看帖子
6. 切到后台，展示图书管理或借阅管理列表

> 若现场网络异常：直接展示流程截图，口述“请求-响应-数据库变化”即可。
